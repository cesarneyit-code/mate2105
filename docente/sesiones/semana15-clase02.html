Preflight técnico y conceptual

Antes de dictar, confirme sin improvisar:

  1. que cada grupo puede descargar el repositorio que revisará;
  2. que la corrida sin ayuda oral tiene un punto de inicio inequívoco;
  3. que un hallazgo solo se acepta cuando incluye pasos para reproducirlo;
  4. que existe una ruta de contingencia para archivos o entornos rotos;
  5. que la devolución no expone notas, soluciones ni comentarios privados.

Frase de apertura

¿Puede otro grupo abrir, ejecutar y cuestionar el proyecto usando únicamente lo entregado?

Después de leerla, confirmar en voz alta qué grupo revisa cada repositorio y qué producto debe devolver.

Trabajo de tablero

Marco de la revisión

  • comportamiento que se quiere sostener: resultado final que un tercero debe poder reconstruir;
  • evidencia observada en la corrida: corrida limpia, manifest y tabla de soporte;
  • condición que permite compararlas: el paquete contiene datos permitidos, entorno y orden de ejecución;
  • variación que la revisión no elimina: muestra, plataforma y dependencias.

Observación que debe quedar explícita: Reproducir una cifra no la vuelve menos aleatoria; prueba la ruta computacional que la produjo.

Regreso al caso: La clínica distingue fallos de ejecución, discrepancias numéricas y límites de inferencia.

Zona izquierda: objetos que no se borran

Recurso Prueba de aceptación
README instala y ejecuta desde cero
datos fuente, licencia, snapshot/hash
entorno versiones reconstruibles
manifest rutas existentes y hashes válidos
reporte cifras iguales a artefactos

Zona central: expresión y derivación

\[ \text{completitud}=\frac{\#\text{controles aprobados}}{\#\text{controles}}, \quad\text{con bloqueos críticos reportados aparte.} \]

La revisión sigue dependencias: README → entorno → datos → ejecución → artefactos → reporte.

Si un paso falla, se registra comando, error, sistema y propietario. No se reemplaza por una impresión general de que ‘casi funciona’.

Zona derecha: síntesis final

  • seguir README sin ayuda
  • registrar comando y error
  • priorizar bloqueos
  • cerrar con nueva evidencia

Respuestas esperadas

Producto de la clínica

Primero hash de datos, luego semilla, después tipografía. Cierre con hash recalculado/fuente confirmada, corrida repetida con semilla y revisión visual final.

El producto mínimo es un backlog priorizado. Cada fallo crítico debe incluir pasos de reproducción, responsable y prueba de cierre.

Salida numérica de referencia

crítico (ejecución/datos), alto (métrica/hash), responsable (por hallazgo).

Errores frecuentes y respuesta docente

Error Respuesta docente breve
Ayudar oralmente al revisor oculta instrucciones faltantes en README. Pedir que localicen esta señal: el autor dicta pasos no escritos.
Marcar todo como aprobado por cercanía de entrega elimina la función de la clínica. Pedir que localicen esta señal: no hay evidencia o comando junto al estado.
Corregir directamente el repositorio del otro grupo. Detener la edición y exigir primero un hallazgo reproducible.

Qué recortar si faltan minutos

No recortar la revisión cruzada, el registro de hallazgos ni la devolución al grupo autor.

Si faltan 10 minutos:

  1. omitir la puesta en común general;
  2. conservar la corrida sin ayuda oral;
  3. exigir un fallo priorizado con responsable y prueba de cierre;
  4. exportar el backlog antes de salir.

Si faltan 20 minutos, revisar solamente el flujo principal del proyecto y programar una verificación asincrónica de los artefactos secundarios.

Contingencias

  • Sin proyector: usar la expresión central, el ejemplo y las cuatro líneas de tablero; el deck no es imprescindible.
  • Notebook no abre: mostrar la salida HTML o capturas verificadas y entregar el archivo después; no improvisar resultados.
  • Entorno roto para varios estudiantes: trabajar en parejas durante la demostración y registrar el error para soporte posterior.
  • Pregunta fuera de alcance: anotar la pregunta y responder después con una referencia o un cálculo, sin inventar una certeza.

Después de clase

Registrar en cinco líneas:

  1. consolidar los hallazgos reproducibles de la revisión cruzada;
  2. separar fallos críticos de mejoras editoriales o cosméticas;
  3. confirmar responsable y prueba de cierre para cada fallo crítico;
  4. registrar los repositorios que requieren una segunda verificación;
  5. publicar únicamente el plazo y el procedimiento de cierre.
