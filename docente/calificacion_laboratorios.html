Calificación de laboratorios

Entrega, autograding, revisión asistida y devolución

Decisión operativa

La ruta principal es Bloque Neón → Gradescope → autograder → revisión manual asistida → publicación de nota. GitHub Classroom queda como espacio opcional de trabajo; nbgrader no se adopta mientras el curso no tenga un JupyterHub y un servicio de intercambio administrados.

Esta decisión minimiza soporte y evita que el dominio desigual de Git se convierta en una barrera de entrega. Gradescope acepta el .ipynb directamente, ejecuta pruebas en escala, permite recalificar todo el lote y puede devolver la nota al LMS mediante LTI.

Configuración inicial

  1. Confirma con la Universidad la licencia de Programming Assignments y el enlace LTI.
  2. Crea una actividad por laboratorio y habilita Manual Grading.
  3. Usa 90% automático en Lab 1 y 85% en Labs 2-5.
  4. Crea una sola pregunta manual con niveles 0, 1 y 2.
  5. Sube el ZIP privado del autograder y prueba solución, incompleto y timeout.
  6. Sincroniza roster, fechas y columna de notas antes de publicar.

Cierre en cinco comandos

Los comandos se ejecutan desde la raíz del repositorio privado.

.venv/bin/python -m gestion_laboratorios preparar lab01 \
  --entregas ~/Downloads/lab01_submissions.zip \
  --roster ~/Downloads/lab01_grades.csv \
  --resultados ~/Downloads/submission_metadata.yml \
  --salida trabajo_calificacion/lab01

.venv/bin/python -m gestion_laboratorios borradores-ia lab01 \
  --run trabajo_calificacion/lab01 --solo-preparar

.venv/bin/python -m gestion_laboratorios borradores-ia lab01 \
  --run trabajo_calificacion/lab01

.venv/bin/python -m gestion_laboratorios panel lab01 \
  --run trabajo_calificacion/lab01

.venv/bin/python -m gestion_laboratorios finalizar lab01 \
  --run trabajo_calificacion/lab01 \
  --revision ~/Downloads/revision_lab01.csv \
  --item-notas "Lab 1"

Antes del segundo comando de IA, configura OPENAI_API_KEY en el entorno. Nunca la guardes en el repositorio. --solo-preparar permite auditar exactamente el contenido seudonimizado que se enviaría.

Revisión en Gradescope

El panel local reúne respuesta, resultados visibles, pruebas fallidas, nivel sugerido y banderas. Aprueba todas las propuestas y dedica lectura detallada a:

  • contradicción entre texto y resultados;
  • notebook inválido o incompleto;
  • puntaje automático inferior al 50%;
  • falta de evidencia o afirmación fuera de alcance;
  • muestra aleatoria de auditoría del 10%.

Después usa la pregunta manual de Gradescope con tres ítems y navegación Next Ungraded. Los niveles se aplican con teclas numéricas; el comentario aprobado se copia desde el panel. Publica en Gradescope y usa Post Grades para el LMS.

Qué hace la IA

La IA no ejecuta código ni modifica el puntaje técnico. Recibe un identificador seudónimo, la respuesta manual, salidas etiquetadas, el resumen de pruebas y la rúbrica. Devuelve JSON estricto con nivel, comentario, evidencia, banderas y confianza. El profesor conserva la decisión final.

El runner omite las celdas etiquetadas evidence al recalificar: esas celdas sirven para la interpretación y conservan las salidas de la ejecución del estudiante. Las funciones y respuestas estructuradas se prueban aisladamente, de modo que un error en el experimento no elimina todo el crédito parcial.

No se envían nombres, correos, códigos, rutas ni notebooks completos. La API no usa datos para entrenamiento por defecto, pero puede conservar logs de abuso hasta 30 días salvo controles aprobados; por eso la seudonimización y la revisión institucional son obligatorias.

Contingencia

Si Gradescope no está disponible, crea una tarea de archivo en Bloque Neón, descarga el ZIP completo y usa el fallback Docker:

.venv/bin/python -m gestion_laboratorios construir-docker
.venv/bin/python -m gestion_laboratorios calificar-local lab01 \
  --entregas entregas_lab01.zip \
  --salida trabajo_calificacion/lab01_resultados

El contenedor corre sin red y con tiempo, memoria, CPU y procesos limitados. No uses --confiar-codigo con entregas reales.
