Proyecto final

Empieza aquí

El proyecto reúne el trabajo aplicado del semestre. Un grupo de 2 o 3 estudiantes formula una pregunta, documenta un conjunto de datos real, construye un baseline, compara modelos y presenta una conclusión cuyo alcance pueda justificarse.

Esta página es el contrato operativo del proyecto. Aquí están los productos, fechas, nombres de archivo, puntajes y lugar de entrega. Si una guía modular resume un hito, esta página conserva la especificación completa. Cualquier ajuste se anunciará en Bloque Neón y se reflejará aquí.

Resultado esperado

El proyecto no premia el modelo más complejo. Premia una cadena completa y verificable:

pregunta -> población y muestra -> estimando -> baseline -> modelos
         -> validación e incertidumbre -> límites -> conclusión

Al final, otra persona debe poder entender la pregunta, reproducir el resultado principal y distinguir lo que los datos permiten afirmar de lo que todavía no está probado.

Contrato del proyecto

Equipos y alcance

  • Los equipos tienen 2 o 3 integrantes. Un proyecto individual requiere autorización previa del profesor.
  • Cada equipo trabaja con un dataset real y una pregunta propia. Dos equipos no pueden presentar el mismo problema con el mismo dataset sin autorización.
  • Se permiten proyectos supervisados, no supervisados o híbridos. El alcance queda aprobado en el hito de Semana 4.
  • La entrega escrita es grupal. La defensa y su calificación son individuales.
  • No se exige publicar el repositorio en internet. El .zip entregado en Bloque Neón debe ser autosuficiente, salvo datos grandes o restringidos debidamente documentados.

Requisitos mínimos

Todo proyecto debe incluir:

  1. una pregunta aplicada y un usuario o contexto de uso;
  2. población objetivo, muestra observada, unidad de observación y mecanismo de inclusión;
  3. fuente, licencia o condiciones de uso y diccionario de los datos;
  4. estimando, estimador, estimación y una fuente relevante de incertidumbre;
  5. EDA y diagnóstico de calidad de datos;
  6. partición de datos justificada y transformaciones sin leakage;
  7. baseline simple y al menos un modelo alternativo comparable;
  8. métrica principal definida antes de seleccionar el modelo;
  9. análisis de errores, residuos, slices o estabilidad, según el problema;
  10. registro reproducible de experimentos;
  11. límites técnicos, poblacionales, éticos o legales;
  12. una conclusión proporcional a la evidencia obtenida.

Reglas de datos

El dataset debe tener fuente verificable, documentación suficiente y tamaño manejable en un computador personal. No se aceptan credenciales, datos personales identificables, material cuya licencia impida el uso académico ni scraping frágil como única forma de reconstrucción.

Si los datos no pueden incluirse en la entrega, data/README.md debe indicar cómo obtenerlos, su versión o fecha de consulta, las transformaciones realizadas y un hash cuando sea posible. El kit incluye esa plantilla.

Calendario de hitos

Todas las horas están expresadas en hora de Bogotá (UTC-5). Salvo la defensa, una sola persona del grupo sube la entrega en:

Ruta de entrega: Bloque Neón → Contenido → Proyecto final → tarea indicada.

La marca de tiempo de Bloque Neón es la referencia. Un enlace de repositorio, un correo o un archivo compartido por fuera de la tarea no reemplazan la entrega. La última versión completa recibida antes del cierre es la que se califica.

Contenido de cada hito

P00: ideas iniciales

Incluye dos ideas dentro del laboratorio diagnóstico. Para cada una: tema o dataset, fuente posible, unidad de observación, población, forma de observación, pregunta, uso de la respuesta y limitación anticipada. No es una aprobación de tema.

P01: lista corta de datos

Entrega un PDF de máximo 2 páginas con 2 o 3 opciones. Para cada opción registra fuente, licencia, unidad de observación, tamaño aproximado, objetivo o pregunta, variables disponibles, riesgo principal y forma de acceso. Cierra con una opción preferida y la razón.

P02: prepropuesta

Entrega un PDF de máximo 2 páginas. Usa la plantilla de hitos y completa pregunta, usuario, población, muestra, unidad, estimando, variables, métrica tentativa, baseline y límite principal.

P03: propuesta aprobable

Entrega un PDF de máximo 4 páginas, sin anexos. Debe fijar dataset, pregunta, protocolo de partición, estimando, estimador, métrica principal, baseline, dos modelos candidatos, análisis de incertidumbre, riesgos y plan de trabajo. El profesor responde Aprobada, Aprobada con cambios o Reformular.

P04: baseline

Entrega un .zip con esta estructura mínima:

grupo##_hito06_baseline/
  README.md
  baseline.ipynb
  data/README.md
  artifacts/metrics.json

baseline.ipynb debe ejecutar de arriba hacia abajo, conservar salidas y mostrar partición, baseline, métrica principal, resultado y una limitación. README.md indica el orden de ejecución. No uses el test set para decidir el modelo.

P05: checkpoint técnico

Entrega un .zip con checkpoint.pdf de máximo 3 páginas, el notebook o código actualizado y artifacts/metrics.json. Compara el nuevo experimento con el baseline usando la misma partición y métrica; incluye variación entre repeticiones o folds, análisis de errores y el siguiente experimento priorizado.

P06: análisis estructural

Entrega un .zip con estructura.pdf de máximo 2 páginas y un notebook ejecutable. Usa PCA, K-Means, recomendación o un análisis estructural alternativo aprobado. Debe contener pregunta, preprocesamiento, resultado cuantitativo, figura o tabla, estabilidad y límite. Si un método del módulo no es pertinente, acuerda la alternativa con el profesor antes del 22 de octubre; no basta escribir “no aplica”.

P07: reproducibilidad

Entrega una copia ejecutable del proyecto, sin .git, cachés ni datos restringidos. Debe contener README.md, requirements.txt, data/README.md, código, artifacts/manifest.json, artifacts/metrics.json y una ejecución limpia registrada. Una ruta local, una credencial o una celda ejecutada fuera de orden impiden aprobar este hito.

P08: borrador final

Entrega el borrador del reporte con todas sus secciones y resultados principales. Puede tener tablas o figuras pendientes claramente marcadas. Se usa para la clínica de Semana 15 y no sustituye la entrega final.

P09: entrega final

El archivo no debe superar 50 MB. Si los datos son más grandes, enlázalos y documenta su recuperación. No uses .rar ni dependas de un repositorio externo como único soporte.

grupo##_proyecto_final/
  README.md
  requirements.txt
  data/README.md
  notebooks/
    01_exploracion.ipynb
    02_modelado.ipynb
  src/
  reports/
    reporte_final.pdf
    presentacion.pdf
  artifacts/
    manifest.json
    metrics.json
  contribuciones.csv

El reporte tiene máximo 10 páginas de cuerpo, sin contar referencias ni un apéndice estrictamente técnico. La presentación tiene 6 a 8 diapositivas de contenido, más referencias o respaldo si se necesitan. El apéndice no reemplaza argumentos que deban estar en el cuerpo.

P10: defensa

Cada equipo dispone de 6 a 8 minutos para presentar tesis, datos, método, resultado y límite. Después, cada integrante responde durante 3 a 5 minutos preguntas sobre su contribución, matemática, validación, reproducibilidad y limitaciones. Consulta la guía de defensa.

Formato común de entrega

Convenciones de archivos

  • Sustituye grupo## por el número asignado en Bloque Neón, por ejemplo grupo07.
  • Los nombres internos usan minúsculas, números y guion bajo; no llevan tildes ni espacios.
  • Todo PDF debe permitir seleccionar texto y tener tablas y figuras legibles.
  • Todo notebook debe ejecutar de arriba hacia abajo en el entorno publicado del curso.
  • Las salidas que sostienen una afirmación deben permanecer visibles.
  • Ningún archivo debe contener contraseñas, tokens, rutas personales o datos identificables.
  • README.md debe listar integrantes, orden de ejecución y resultado esperado.

Qué ocurre si falta un archivo

La calificación usa solamente lo que puede abrirse desde la entrega recibida en Bloque Neón. Un archivo faltante, corrupto o dependiente de una ruta local se evalúa como evidencia ausente en el criterio correspondiente. Antes de enviar, descomprime el .zip en una carpeta nueva y sigue tu propio README.md.

Cinco criterios reciben 0, 1 o 2 puntos: autoría, matemática, validación, reproducibilidad y limitaciones. Los descriptores completos están en las rúbricas del proyecto.

Preguntas frecuentes

¿Debo usar un dataset del curso? No. Los datasets del curso sirven para practicar. El proyecto puede usar otro conjunto real si cumple las reglas y se aprueba.

¿Necesito GitHub? No. Puedes usarlo, pero el .zip de Bloque Neón es la entrega oficial. Un repositorio privado debe dar acceso al profesor y no sustituye el archivo.

¿Debo usar una red neuronal? No. El modelo debe responder a la pregunta y compararse de forma justa con el baseline.

¿Puedo cambiar de tema? Sí, hasta la propuesta de Semana 4. Después se requiere autorización porque el cambio afecta los hitos ya evaluados.

¿Qué hago si los datos son privados o muy grandes? No subas datos sensibles. Entrega un data/README.md reproducible y acuerda con el profesor una forma segura de verificación antes de la propuesta.

¿Dónde consulto un cambio de fecha? En la tarea correspondiente de Bloque Neón y en esta página. El calendario mantiene la vista resumida del semestre.
